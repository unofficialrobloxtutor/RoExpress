Origin

The Story

RoExpress started as a personal utility in 2021 and grew into a full networking framework over five years. Here's what actually happened.

2021 — The problem
Every game had the same boilerplate

Building Roblox games meant writing the same code over and over — FireServer, OnServerEvent, check the player, check the data, respond somehow. There was no standard. Some games used string keys, some used event names, some used argument position. None of it composed.

Express.js had already solved this problem for web APIs a decade earlier. Routes, params, middleware, status codes — a uniform interface that every developer on the project could understand immediately. The only question was whether the same model would work on Roblox.

2022 — First version
v1: a thin wrapper, mostly for personal use

The first version was a single ModuleScript — a few hundred lines. It put a string-keyed router on top of RemoteEvent, added per-player rate limiting, and nothing else. It wasn't published. It was just used in personal projects.

The immediate win was structural: other team members could read a route handler and understand it without knowing how the underlying remote worked. That was the signal that the model was right.

2023 — HTTP semantics
Adopting GET/POST/PUT/DELETE instead of string event names

The first version used arbitrary string names for handlers. Renaming a handler meant searching the entire codebase. HTTP verbs solved this — a GET handler is always read-only, a POST always creates or triggers, a DELETE always removes. The semantics were self-documenting.

Status codes followed from the same logic. Returning 404 for a missing resource was more informative than returning false or an empty table. Clients could pattern-match on the status without parsing the response payload.

2024 — The v2 rewrite
Ground-up redesign — all nine modules, Codec, Stream, TypeCoercer

By mid-2024 it was clear the v1 architecture couldn't extend cleanly. Compression, typed params, server push, internal events, high-frequency state sync — each needed its own surface. The decision was to do a complete rewrite rather than bolt things on.

v2 shipped nine modules: App, Network, Listener, Broadcast, Bridge, Stream, Codec, Router, TokenBucket, TypeCoercer. Each was built to be used independently — you don't need all nine to get started.

2025 — Push, Tamper, Ports, Benchmark
The missing pieces — reliable push, exploit detection, isolation, profiling

v2.1 and v2.2 filled the gaps that became obvious once people used v2 in real projects. Reliable server push turned out to be the most-requested feature — handling it on the same RemoteEvent as request-response kept the remote count down and made the Listener transparent.

Tamper came from watching how games deployed under load. Players would probe routes with malformed payloads to find edge cases. A built-in strike tracker handled the common case without requiring every handler to write its own defence.

2026 — Stream v3 & compact args
v2.3: Stream rebuilt for any data, not just FPS

The original Stream was purpose-built for FPS player position sync. v2.3 rebuilt it as generic typed channels — you define a schema, Stream handles the packing. The FPS use case still works, but now you can stream wheel rotation, weather state, economy ticks, or anything else that changes every frame.

Compact handler args shipped alongside — four arguments shrinking to two was the most common ergonomics complaint. The detection approach (checking parameter count via debug.info at dispatch time) let both forms coexist without a migration.

By the numbers: 9 modules  ·  ~1800 lines of source  ·  5 years from first version to v2.3
